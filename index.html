Software Challenge

For this task we have to develop the basic elements of a software that is calculating the correct dosage of a medicine, which is prescribed by a vet doctor. This software has to take into consideration different factors such as the dog's age, weight and nursing status in case of females and also generate the final cost of the prescription.

  1. Application requirements:

    FUNCTIONAL NON - FUNCTIONAL
    Secure login for authorised staff. The system should be available whenever the clinic is open.
    Maintain a record of previous prescriptions. The scalability of such project should be considered, thus being able to add more storage resources should be easy.
    An alert system that is notifying the user about over or underweight animals. Must be compatible with most browsers and devices, since the clinic may use computers or tablets to access it.
    Calculation of the appropriate dosage depending each animal's weight, age (when it is a puppy) or nursing status (in case of females). The interface must be user friendly, but at the same time to fit to a vet clinic. Graphics should be carefully be picked.
    Calculation of the cost, depending on the type and quantity of the medicine. The software has to be documented so any new doctor who starts working, could understand every feature to avoid mistakes.
    Being able to see and edit the information / medical history of the animal (such as breed, age, weight). The patient's database should backed up more than 2 times a day to avoid any missing files.

  2. Sequence of instructions:

    1. The user logs through the login page by using the valid credentials.
    2. The user either creates a new patient's form or searching for existing ones by using the menu or the search box accordingly. Through the form, the user will add or modify the animal's information, such as breed, chip ID, weight, colour, previous health history/incidents, nursing status and age.
    3. Now that the system has all the information needed for the calculations, the user using the menu can use the dosage calculator.
    4. By checking the correct medicine, the system calculates the dosage needed for this specific patient based on the information that the user previously saved. At this stage, the medicines cost is also calculated.
    5. The system generates an analytical prescription including any additional medicine may needed (in case of nursing for example) such as tapeworm drops, but also includes the cost of each medicine, the quantity and any extra information such as "usage instructions" by the user or alerts depending the breed, weight and the age of the individual animal. Now the user has the option to send this prescription to be printed as the system have already saved the prescription into the patient's history.
    6. The user can logout and turn off the system or just close the browser window.

  1. Rewrite your Use case for check balance, using the more Detailed Use Case template.



    Case#0001

    Case name: Check balance to withdraw cash
    Actors
    • primary actor: user (account owner)
    • secondary actors: bank system, ATM machine
    Description The user want to verify his available credit in his bank account and based on the outcome, to be able to withdraw cash. Such interaction involves the user initiating a transaction on the ATM using his card and PIN to get verified, which then communicates with the bank's servers to retrieve the information. The outcome is a secure process where the user get informed about the credit balance and if sufficient funds are available, then proceed to withdraw the cash.
    Trigger The user is ready to go for shopping or maybe a long night out with friends on a pub. Before he start spending, to be able to feel safe about his remain credit, he has to check if there are any available funds in the account. Since there are times where a blackout or and internet disruption can take place, having some cash in the wallet is critical, even to get a cab later to return at home.
    Preconditions
    • The user always have the bank card inside the wallet.
    • The user know his PIN.
    • ATMs can be found around his area at all times and is functioning normally.
    • The user has a sufficient balance in his account.
    Postconditions (guarantees)
    Success guarantees:
    • The ATM will display the available balance.
    • If the user choose to withdraw cash and there is sufficient balance, he can request a specific amount of cash to get.
    • After his successful initiation with the ATM the user will receive a confirmation of his transaction.
    Minimal guarantees
    • The ATM will terminate the initiation, even if the user do not have sufficient balance for the user to withdraw, so other users could use the system.
    Normal Flow
    1. The user put the card on ATM
    2. The ATM check the card and asking for PIN verification
    3. The user inputs the PIN
    4. The system validates the PIN. The PIN is valid
    5. On screen a menu is presented
    6. The user choose to check the balance
    7. The ATM gives the option to print the balance on screen or on paper
    8. The user pick the screen method
    9. ATM showing the available credits on screen and then asks if the user wants to withdraw, to return on menu or to log out
    10. The user choose to withdraw 50 euros
    11. ATM validates that the requested amount is not more than the available credits and starts counting the cash
    12. ATM drop out the card and wait for the user to remove it while a thanks / goodbye message show on screen
    13. The user removes the card. The ATM present the cash. The user takes the cash and leaves
    14. ATM screen refreshes and goes back to the intro screen where a video is playing
    Alternative Flows C1) The user miss-typed an invalid PIN. The ATM prints an error message on screen and giving another chance to the user to input the correct PIN. This is a loop that expecting from the user either the correct pin or to cancel the transaction.
    C2) The user typed 3 times an invalid PIN and then the ATM will hold the card and prompt the user to contact the bank.
    G1) The user choose to receive the balance output printed on receipt. The ATM generate a receipt displaying the available credits and details such as date/time, address of ATM.
    I1) The user choose to logout. The ATM ends the transaction, drop out the card and display a "Good bye" message on screen.
    J1) If the user attempt to withdraw an amount that exceeding the available credits the ATM will display an error message prompting the user to input a smaller amount or to cancel the transaction.
    J2) The user choose an amount for example 50euros but this specific ATM works only with 20e and 50e bills. The ATM will display an error message prompting the user to input a different amount or to cancel the transaction.

    a. Specify some tests that could be used from each area of the PACT.

    Actors
    • Verification:We will verify that individual modules and functions of the software will work correctly. Then we will test if the modules will work together seamlessly.

    • Verification:We will test the application on different browsers and devices to check its compatibility. Also we will apply security testing to make sure that data protection and authentication mechanisms are effective.

    • Verification:We will verify the compatibility across different technologies by checking its behaviour through the cross browser and cross device testing.
